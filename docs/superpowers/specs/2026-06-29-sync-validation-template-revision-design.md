# 真机同步验证模板修订设计

## 目标

把现有真机同步验证任务包改成可执行、可复现、不会在未确认时清除用户数据的最小流程，并让正式模板进入 Git 管理。

## 方案比较

1. **推荐并采用：正式模板迁入 `docs/`，原路径保留跳转说明。** 正式内容可版本管理，历史调用路径仍能找到新文件。
2. 只修改 `review-artifacts/_templates/`。改动最少，但整个目录被 Git 忽略，无法提交和长期维护。
3. 两处各保留完整副本。兼容性最好，但后续容易出现两份规则不一致。

用户已选择方案 1。

## 正式模板结构

正式文件为 `docs/agent-task-sync-validation.md`，保留以下主链路：

1. 读取 HEAD、版本和设备状态，确认 tracked 工作区干净。
2. 运行 typecheck、构建 APK，并从实际生成路径读取元数据和固定签名。
3. 在任何 `pm clear` 或卸载前检查备份，并要求用户明确确认。
4. 逐类创建最小测试数据并完成冷启动检查。
5. 等待所有 outbox push 成功后，只做一次清数据或重装。
6. 登录同一老账号，集中验证所有实体从云端恢复。
7. 输出 Markdown 摘要和 JSON 结果，记录证据、未覆盖项及测试数据 ID。

## 删除与调整

- 删除额外衣物 B/C、每类数据重复卸载、旧测试脚本专项清点和可选 `VERSION_HISTORY` 修改阶段。
- MiniMax Key 移到本机存储附加检查，不计入云同步结论。
- 草稿恢复等非同步测试移到可选附录。
- 修正 macOS 版本读取、pipeline 退出码、APK 实际路径和 logcat 时间边界。
- 报告标题统一为“老账号新设备同步验证”，手机号默认脱敏。
- 测试目录加入时间戳，避免同版本复测覆盖。

## 安全与失败规则

- 缺签名文件、APK 元数据不符、tracked 工作区不干净：停止。
- typecheck、构建或安装失败：记录后停止进入破坏性阶段。
- push 未确认成功：不得清数据或卸载。
- 清数据、卸载、使用个人参考照、清理云端测试数据：必须单独取得用户确认。
- 不读取、记录或回显密码和 MiniMax Key。

## 验证

- 检查模板内不再出现失效的 `sed` 版本提取、错误根目录 APK 假设和重复卸载流程。
- 检查 Markdown 结构、示例命令和 JSON 字段一致。
- 运行 `git diff --check`；本次仅文档变更，不运行应用测试。
