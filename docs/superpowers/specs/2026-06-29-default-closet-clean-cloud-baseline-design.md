# 默认衣橱与云端全新基线设计

## 目标

1. 每个新账号固定拥有一个默认衣橱。
2. 默认衣橱在本地立即可用，云端通过 outbox 后台持续同步，无网不阻塞进入 App 或本地录入。
3. 默认衣橱不可改名、编辑或删除，作为衣物和自定义衣橱删除时的最终兜底位置。
4. 清空 `http://111.231.98.86` 当前所有线上数据和用户文件，不保留历史账号、会话、认证、诊断或同步数据，不做旧数据迁移。

## 默认衣橱规则

- 稳定业务标识：`dexieId = "home"`。
- 固定名称：`默认衣橱`。
- 固定简介：`默认衣橱`。
- 固定排序：`sortOrder = 1`。
- 新账号首次建立本地 workspace 时创建一次，同一本地 workspace 不重复创建。
- 同步不作为进入 App 的门禁；失败记录保留在 outbox，由现有重试机制续传。
- 设置页和全部衣橱页中，默认衣橱行不打开编辑弹窗，仅展示“默认”标识。
- 位置更新和删除命令层对 `home` 直接拒绝，不生成删除墓碑或 outbox。
- “清空数据”保留默认衣橱；自定义衣橱可删除，其中衣物可迁移到默认衣橱。

## 衣橱同步契约

- 在 cloud contracts 中增加衣橱专用 schema，不再只使用泛型 `SyncEntitySchema`。
- 衣橱 payload 明确要求 `dexieId`、`name`、`note`、`sortOrder`；客户端 push、服务端保存和 bootstrap 返回都使用同一契约。
- 默认衣橱的固定字段由客户端命令层产生，服务端契约拒绝缺字段的新写入，防止再出现空 payload 衣橱。
- 本次不恢复、匹配或迁移历史衣橱；云端清空后只验证全新账号。

## 生产清空方案

目标服务器：`111.231.98.86`。

保留：

- Docker Compose 部署、PostgreSQL 数据库和表结构。
- Drizzle 迁移记录。
- JWT/诊断密钥、Caddy 和服务器配置。
- `/srv/wardrobe/storage` 根目录本身。

永久删除：

- PostgreSQL 业务表中的所有行：用户、手机号身份、密码凭据、待完成注册、设备会话、refresh token、安全事件、衣橱、衣物、套装、种草、穿着事件、计划、画像、资产、同步变更/变更请求、诊断工单/审计/请求轨迹。
- `/srv/wardrobe/storage/users` 下的全部用户图片和诊断文件。
- 现有数据库备份不新增；用户已明确要求不保留旧数据。

执行顺序：

1. 记录 API 版本、数据表行数和 `/srv/wardrobe/storage/users` 文件数/容量，不读取密码、token 或用户文件内容。
2. 停止 `wardrobe-api` 容器，防止清空期间产生新写入。
3. 在 PostgreSQL 事务中清空全部应用数据表并重置序列，不删表、不删数据库。
4. 逐项核对后永久删除 `/srv/wardrobe/storage/users` 下用户文件，保留 storage 根目录。
5. 重启 API，检查 health/ready/version。
6. 复查所有应用表行数为 0，storage 用户文件为 0，旧账号无法登录。

在真正执行第 3、4 步前，必须将数据库目标和文件路径再次展示给用户，获得对具体目标的最终确认。

## 实施与验证顺序

1. 先实施默认衣橱规则和专用同步契约。
2. 运行客户端 typecheck/逻辑/build，以及 contracts/API typecheck 和 API 测试。
3. 部署新 API，确认 health/ready/version。
4. 获得删除目标的最终确认后执行生产清空。
5. 新建账号，断网验证本地默认衣橱可用；联网后确认 outbox 成功上传。
6. 再次清除客户端本地数据并重登，确认云端恢复的默认衣橱字段完整。
7. 验证默认衣橱无编辑/删除入口，命令层拒绝更新/删除，清空数据后仍保留。

## 风险

- 生产清空不可恢复，且按用户要求不创建旧数据备份。
- 旧 APK 仍可能携带本地 token 或旧 workspace；服务器清空后 token 失效，旧本地业务数据不应自动并入新账号。
- 本轮不为旧数据增加迁移、合并或修复分支。
