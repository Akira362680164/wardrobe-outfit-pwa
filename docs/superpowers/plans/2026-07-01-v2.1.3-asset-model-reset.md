## 1. 审查结论与修订

ChatGPT 对根因的判断正确，当前代码确实存在以下断层：

- 正式领域类型仍保存 `imageDataUrl`、`thumbnailDataUrl` 等图片内容。
- Overview Mapper 会下载 thumbnail，并把它填入正式对象的原图字段。
- Repository 使用 `WeakMap<object, metadata>` 保存 entityId、revision 和资产引用，对象展开后元数据丢失。
- 服务端资产直接归属单一业务对象，种草转衣橱通过转移 owner 实现，无法共享引用。
- `readWorkspaceTryOnProfile()` 固定返回默认值，Data Controller 未接入 Overview 中的画像。
- `renderContent` 使在线图片绕过共享轮播轨道。

原任务书以下部分改为本计划口径：

1. 不从公开仓库提交 `68d9079` 拉分支；以本地最新 `f421bf7` 为唯一开发基线。
2. 不启动任何 subagent；所有阶段由母 Agent 串行开发、验证和提交。
3. 不给所有可编辑领域类型直接强加必填服务器字段。新增明确的服务器实体元数据，并区分服务器读回对象与本地创建草稿，避免默认画像、新建衣橱位置等未保存对象伪造 server ID。
4. 不预设新增 `use-server-entity-state.ts`、`use-entity-write-lock.ts` 等抽象。先复用现有 Controller、Repository 和异步按钮；只有两个以上真实调用点需要共享逻辑时才抽取最小纯函数或 Hook。
5. 图片请求去重不能让单个组件的 AbortSignal 取消其他组件共享的请求。共享下载只按 `assetId + variant + sha` 去重；组件卸载通过订阅释放和 stale token 防止写回，最后一个引用释放后才撤销 Object URL。
6. 测试服务器清理前先在内存中收集所有图片和诊断对象 storage key；数据库事务提交后按该清单删除存储对象。删除失败写入报告并重试，不能先删数据库后丢失对象定位信息。
7. 不要求人为凑足固定提交数；以下八个阶段默认各形成一个可独立验证的提交，若某阶段无代码变化则不制造空提交。

## 2. 全局不可变约束

- 正式业务对象和 API payload 不保存 Data URL、Base64、Blob URL 或 `local*` 字段。
- 本地未提交图片只存在当前页面内存，失败后保留；草稿未变时复用 `clientMutationId`。
- Overview 只返回业务结构、服务器元数据和资产引用，不请求图片内容。
- 图片正式保存顺序为临时会话、上传 variants、业务事务绑定、服务器读回。
- 写成功后以服务器返回完整对象替换状态，不使用旧对象与本地 patch 合并。
- 不增加持久缓存、离线队列、乐观写入、后台同步、新依赖或新 AI 功能。
- 不迁移测试服务器旧业务数据，不保留旧字段双读、双写或兼容回退。
- 文件删除严格执行 `AGENTS.md`：删除前后检查状态，目标逐项列出并移入废纸篓。
- 每阶段更新 `VERSION_HISTORY.md`，记录 high 风险门禁、验证和“未触发 subagent：用户明确要求母 Agent 串行开发”。

## 3. 目标数据边界

### 3.1 正式读回类型

在 `src/lib/types.ts` 定义并复用：

```ts
export interface ServerEntityMetadata {
  serverEntityId: string;
  serverRevision: number;
}

export type ImageAssetVariant = "original" | "thumbnail";

export interface ImageAssetReference {
  assetId: string;
  variants: ImageAssetVariant[];
  mimeType: string;
  width?: number;
  height?: number;
  sha256: string;
  variantSha256?: Partial<Record<ImageAssetVariant, string>>;
}

export interface CroppedImageReference {
  asset: ImageAssetReference;
  cropBox?: GarmentCropBox;
}
```

服务器读回的衣物、种草、套装、位置、计划和已保存画像必须具有 `ServerEntityMetadata`。本地创建输入继续使用独立 Draft/Input 类型，不伪造服务器元数据。

正式图片字段：

- `WardrobeItem.mainImage?: CroppedImageReference`
- `WishlistItem.mainImage?: CroppedImageReference`
- `SavedOutfit.coverImage?: CroppedImageReference`
- `ReferenceOutfitImage.image: CroppedImageReference`
- `OutfitRealImage.image: CroppedImageReference`
- `TryOnProfile.fullBodyImage?: ImageAssetReference`
- `TryOnProfile.faceImage?: ImageAssetReference`

### 3.2 本地草稿

录入和图片编辑草稿只使用：

```ts
localOriginalDataUrl?: string;
localThumbnailDataUrl?: string;
localCroppedPreviewDataUrl?: string;
localCropBox?: GarmentCropBox;
```

### 3.3 服务端资产

- `assets` 保存一个 canonical asset 的 original/thumbnail storage key 和各 variant 元数据。
- `asset_bindings` 是永久业务引用的唯一事实来源。
- 唯一约束：`userId + ownerEntityType + ownerEntityId + fieldName`。
- 索引：`userId + ownerEntityType + ownerEntityId`、`userId + assetId`。
- `assets.orphanedAt` 只标记无 binding 的正式资产；超过 24 小时后才回收。
- 临时资产继续使用 session、mutation、variant 和 expiresAt，不借用永久 binding。

## 5. 服务器清理算法

```text
确认目标=test且三重口令正确
→ 停止 API 和所有测试写入
→ 查询并保存表计数
→ 查询 assets 与 diagnostics 的 storage keys
→ 数据库事务删除全部用户级数据
→ 提交事务
→ 逐个删除已保存 storage keys
→ 重试失败 key
→ 再查数据库计数
→ 对每个 storage key 执行 stat 确认不存在
→ 输出不含账号、Token、路径和原始 key 的 JSON 报告
```

数据库失败时事务回滚且不删 Storage。数据库成功、Storage 部分失败时数据库保持空，报告列出失败 key 哈希，重复运行 cleanup-only 阶段直至归零。
