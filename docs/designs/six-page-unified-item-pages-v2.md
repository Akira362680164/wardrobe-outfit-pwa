# 6 个页面统一底层框架与页面差异设计（v2 草案，替换 four-page-unified-shell-v1）

**状态**：设计稿，未实现；本稿为对 `docs/designs/four-page-unified-shell-v1.md` 的**整体替换**。
**范围**：6 个页面 = 2 个录入 Step 3 确认页（衣橱 + 种草）+ 4 个 detail/edit 页（衣橱详情、衣橱编辑、种草详情、种草编辑）。
**目标**：
- 把 v1 的 4 页面统一扩展为 6 页面，把 2 个录入 Step 3 确认页纳入"统一底层"。
- 严格区分 **AI 置信度胶囊** 的归属（只在 Step 3）与 **待确认字段胶囊** 的触发条件。
- 继承 v1 的大部分"不重做 UI 风格、不抽太多"原则，但补齐 v1 在录入页与详情/编辑页之间未明确处理的差异点。
- 不改 Dexie schema、不改 MiniMax prompt、不打 APK、不做算法承诺。

## 1. 信息架构总览

### 1.1 6 个页面分两类

| 类别 | 页面 | 入口 | 主要任务 | 是否含 AI 置信度胶囊 |
|---|---|---|---|---|
| **A. 录入 Step 3 确认页**（intake-confirm） | P1 衣橱单品录入 Step 3 | 衣橱首页 → 添加单品 → 步骤 3 | 校对 AI 草稿，确认后保存 | **是**（整件级，0-100 数字 + 颜色） |
| | P2 种草单品录入 Step 3 | 种草页 → 添加种草 → 步骤 3 | 校对 AI 草稿，确认后保存 | **是**（整件级，0-100 数字 + 颜色） |
| **B. 详情/编辑页**（detail/edit） | P3 衣橱单品详情 | 衣橱首页 → 卡片 → 详情 | 只读浏览，底部操作（推荐/试穿/编辑/更多） | **否**（不显示） |
| | P4 衣橱单品编辑 | P3 → 编辑 | 改字段、保存 | **否**（不显示） |
| | P5 种草单品详情 | 种草页 → 卡片 → 详情 | 只读浏览 + 买前评估 + 底部操作 | **否**（不显示） |
| | P6 种草单品编辑 | P5 → 编辑 | 改字段、保存 | **否**（不显示） |

> 命名约定：下文用 `IntakeConfirmShared` 覆盖 P1+P2；`ItemDetailShared` 覆盖 P3+P5；`ItemEditShared` 覆盖 P4+P6。

### 1.2 三类页面的关键差异

| 维度 | 录入 Step 3（P1/P2） | 详情页（P3/P5） | 编辑页（P4/P6） |
|---|---|---|---|
| 字段是否可改 | 可改 | 只读 | 可改 |
| 是否含 AI 置信度胶囊 | **是**（整件级） | 否 | 否 |
| 是否含"待确认"字段胶囊 | **是**（按 4 条规则打） | 仅"待完善"提示（无 AI 标签） | **是**（同 Step 3，按 4 条规则打） |
| 是否有图片头 | 缩略图条（多件） | 主图 3:4 + meta 行 | 主图 3:4 + 重新裁切/识别按钮 |
| 底部操作 | 上一步 / 保存 N 件单品 | 穿搭配推荐 / AI 试穿 / 编辑 / 更多（衣橱） | 顶部保存 + 底部 in-flow 保存 |
| 业务独有字段 | 衣橱：位置/状态；种草：状态(3 态) | 衣橱：位置/购买日期/状态；种草：状态/评估 | 衣橱：位置/购买日期/状态；种草：状态(3 态) |

## 3. 标签规则（重要修正）

### 3.1 "AI 置信度"胶囊

- **唯一归属**：P1 衣橱录入 Step 3、P2 种草录入 Step 3 的"校对草稿"section 标题行右侧，与"待确认 N"横排。
- **数据源（第一期）**：整件级 `confidenceScore: number`，0-100，由 `processAllImagesForRecognition` / `analyzeWishlistIntakeImageOnDevice` 完成后统一计算（**沿用 v1 数据流，不动 prompt**）。
- **阈值**：
  - `>=75` 绿色（moss：`bg-moss/10 text-moss`）
  - `50-74` 黄色（clay：`bg-clay/10 text-clay`）
  - `<50` 红色（denim 红，覆盖现有 `bg-denim/10 text-denim` 太弱，**新建 `bg-clay-strong/10 text-clay-strong`**，或在现有 `clay` 基础上加深透明度）
- **文案**：纯数字 + "AI" 前缀：`AI 86` / `AI 62` / `AI 38`，**不写"高/中/低"**。
- **无值处理**：未走 AI（如用户手填新建，无草稿）→ **不显示**该胶囊；只显示"待确认 N"（N 可为 0）。
- **不归属**：P3/P5 详情页、 P4/P6 编辑页。**禁止**把 AI 置信度胶囊设计进这 4 个页面（v1 的错位 v2 明确纠正）。
- **字段级置信度**：本期**不实现**。如果未来要做"逐字段置信度"（如"主色 90 / 细分 65"），需要 AI 输出结构 + UI 同步支持，列为 v2+ 增强，**不作为第一期实现前提**。

### 3.2 "待确认"胶囊（字段级）

- **归属**：P1/P2/P4/P6 的字段右上角；P3/P5 详情页**不显示**该胶囊（用 §3.3 替代）。
- **触发条件**（**只限 4 条**）：
  1. **低置信**：字段 `confidence === "low"` 或 `needsReview === true` 且 reason 非空。
  2. **识别失败**：AI 未能产出该字段（如"颜色未识别"），`needsReview: true`。
  3. **必填缺失**：该字段对当前 flowKind 属于必填但为空（参见 `requiredFields` 映射）。
  4. **归一化失败**：字段归一化报错（如老 5 字段迁移到 v2 `ColorInfo` 失败、子分类反查 catalog 失败）。
- **不触发**（避免乱打）：
  - 可选字段为空（价格/链接/材质/版型说明/购买日期），**不打待确认**。
  - 用户主动清空（`source === "user"` 且 `needsReview === false`），**不打**。
  - 季节/风格/颜色模式**非空**但子项为空（如季节选了"春"但没选其他）——**不打**（多选字段，0 个也是合法值）。
- **样式**：`bg-clay/10 text-clay` + `rounded-full px-2 py-0.5 text-[10px] font-semibold`，沿用 `FieldSourceBadge` 现有 tone。

### 3.3 详情页"未完善"提示（替换旧"待确认"）

- P3/P5 详情页**不显示** AI 置信度胶囊，也**不显示**"待确认"胶囊。
- 仅当字段**完全为空**且**非可选**（如"材质"可选则不打），在该字段 value 处显示灰字"未填写"，字号 13sp，色 `text-ink/35`。
- 这是详情页的**弱提示**，不打断阅读节奏。
- 评估信息卡（种草独有）若 `WishlistAssessment` 不存在，则在卡内显示"还没有生成买前评估"（沿用 v1.1.22 现状）。

### 3.4 删除项（明确清单）

| 标签 | 是否保留 | 替代 |
|---|---|---|
| `默认` | **删除** | 用户已修改或 AI 给出值的字段**不显示任何标签**；"AI 86" 整件级已足够 |
| `已修改` | **删除** | 同上 |
| `需要留意`（整段 section） | **删除** | 字段级"待确认"已分散到各字段；不再有整块提醒列表 |
| `待确认`（仅 P1/P2/P4/P6） | 保留 | 字段级胶囊，按 §3.2 4 条规则 |
| `待完善`（仅 P3/P5） | 新建 | 详情页字段为空的灰字提示，按 §3.3 |
| `AI 86`（仅 P1/P2） | 保留 | 整件级胶囊，按 §3.1 |

## 4. 字段展示规则

### 4.1 颜色

**Step 3 / 编辑页**（一致）：
- 顶部：颜色模式 3 选 1 chip（**单主色 / 拼色 / 主辅色**），与截图一致。
- 中部：主色 chip grid（`黑/白/灰/蓝/牛仔蓝/棕/米/红/粉/绿/黄/紫`），3 行 ×4 列；当前选中加 ✅ 勾。
- 模式 = `main_with_accent` 或 `multicolor` 时：辅色 chip grid 同样 3×4，多选。
- "展开更多 +N" 按钮收折多余色卡（与 v1 衣橱编辑一致）。
- **UI 校对入口清晰化**（回应"颜色识别错"问题）：
  - 颜色模式默认 = `single` 时，**额外**加一行小灰字提示"识别为单色？若图中含明显拼接/撞色，可改"主辅色"或"拼色""。
  - 识别后 AI 给出的 `primary` 如果命中"黑/白/灰"等通用色，**额外**给一行"AI 识别为 X；如明显不符请改"。这两条小灰字属于"AI 自陈"而非"需要留意"，不触发待确认胶囊。
- **算法承诺**：本设计**不承诺**修复颜色识别错误，仅提供更清晰的校对入口（更长滚动、辅色加可多选、模式可改、提示文案）；实际识别准确率提升由后续 prompt / 模型优化解决。

**详情页**：
- 主色：`ColorChip` + 中文 label（如"白"）。
- 辅助色：`ColorChipList`，多 chip 横排，**不**显示"未识别"字面量（识别失败时显示空 chip + 灰字"未识别"）。
- 颜色模式：中文 label（`单色 / 主辅色 / 拼色`），与 §4.4 一致。

### 4.2 分类 + 细分

- Step 3 与编辑页**完全一致**：一级 chip 横排（9 个：上衣/裤子/半身裙/连体装/鞋/包/帽子/首饰/配饰），二级 chip 多行横排（细分库反查，`T 恤/POLO 衫/衬衫/...`），可横滑。
- 一级切换会触发二级动态刷新（沿用现状）。
- 细分 chip 行超出 4 行时**显示"展开更多"**，避免横向隐藏过深。
- 详情页：只读 `ItemRow`，value 用中文 label（通过 `subcategoryLabel` 反查 catalog id，**不显示** catalog id 字符串）。
- "细分识别错"问题：UI 层面只把入口做清楚，**不承诺**算法修复。

### 4.3 版型倾向（fitGender）

- **详情页**：`ItemRow` value 必须是**中文 label**：
  - `menswear` → `男装版型`
  - `womenswear` → `女装版型`
  - `unisex` → `中性版型`
  - `unknown` → `未识别`（灰字）
- **Step 3 / 编辑页**：4 选 1 chip，**直接显示中文**（截图已是中文，沿用）。chip 选中态用 `bg-denim text-white`，与现状一致。
- 现状问题：衣橱详情 `InfoTab` 当前直接显示 `item.fitGender`（英文字符串），**本期必须修**。

### 4.4 颜色模式

- 详情页 value 必须是中文 label：
  - `single` → `单色`
  - `main_with_accent` → `主辅色`
  - `multicolor` → `拼色`
- 缺失时显示灰字"未识别"。

### 4.5 季节 / 风格

- 详情页：中文 label `/` 分隔（如"春 / 秋"、"休闲 / 甜美"）；空时显示灰字"未识别"。
- Step 3 / 编辑页：多选 chip，已实现。

### 4.6 材质 / 版型说明 / 备注

- 详情页：`ItemRow` value，多行允许换行（最多 6 行省略）。
- 备注长度上限 200（v1.1.22 沿用）。
- 版型说明长度上限 40（v1.1.22 沿用）。

### 4.7 温度

- 详情页：`<TemperatureRangeBar/>`，只读渐变条，数值显示 `minC - maxC`。
- Step 3 / 编辑页：`<TemperatureRangeSlider/>`，双端点可拖，AGENTS.md 触摸规则适用（仅按住滑块拖）。

### 4.8 正式度 / 保暖度

- 详情页：`ItemRow` value = `2/5`。
- Step 3 / 编辑页：`<FormalityWarmthStepper/>` 1-5 加减器。
- 缺失/AI 未识别时：详情页灰字"未识别"；Step 3 字段级"待确认"**不**触发（属于可选数值，非必填）。

### 4.9 价格 / 商品链接

- **永远不打"待确认"**（属于可选字段）。
- 详情页：value 缺失时灰字"未填写"（价格"未填写"，链接"未填写"）。
- Step 3 / 编辑页：placeholder "非必填，例如 399" / "非必填，商品链接"，沿用现状。

### 4.10 衣橱位置 / 状态

- 衣橱位置：衣橱 Step 3/编辑 select 来自 `locations`；详情页只读。
- 衣橱状态：4 态（可穿 / 借出 / 洗涤 / 不可穿）select。
- 种草状态：3 态（感兴趣 / 不感兴趣 / 归档）select。

### 4.11 种草评估 / 购买状态

- 评估信息卡**只**在 P5 种草详情页显示。
- 评估卡内容：`WishlistAssessment.summary` + `scenes` + `pairingTips` + `avoidTips` + `generatedAt` + 顶部"基于 AI 评估"小灰字（已实现于 `DetailAiCard`）。
- **不**在编辑页 P6 显示评估（编辑页只改基础字段；评估由"生成评估"按钮触发，独立于编辑流程）。
- "加入衣橱"按钮在 P5 详情页底部；点击后走 v1.1.22 已实现的 `handleConfirmConvert` 流程。

## 9. 风险点（v2 修正后的新风险）

1. **AI 置信度数据源不统一**：衣橱录入走 `processAllImagesForRecognition` 已有 confidence 字段；种草录入走 `analyzeWishlistIntakeImageOnDevice`。两套 JSON 结构需要校验都能产出 `confidence: number`。**风险**：可能有一边缺字段。**缓解**：阶段 1 实施前先在两套识别路径上各跑一次本地解析脚本确认字段存在。
2. **AI 置信度对种草不一定有现成位置**：`WardrobeItem.aiConfidence?: number` 已有；`WishlistItem` 没有 `aiConfidence` 字段。v2 仅在 Step 3 显示**录入时的草稿 confidence**，不存到种草实体上——与 v1 风险 9 "不动 schema" 保持一致。**风险**：草稿 confidence 生命周期依赖 `IntakeDraft.confidence`；阶段 1 需要保证从识别到 Step 3 之间 confidence 不被丢失。
3. **详情/编辑页"未完善"灰字 vs 字段级"待确认"**：v2 必须保证 P3/P5 详情页**绝不**误用 `review-pill`（字段级待确认胶囊），否则破坏"详情无 AI 标签"规则。**缓解**：把 `review-pill` 写在 `item/` 目录下，并在 PR diff 注释里标注"仅用于 P1/P2/P4/P6"。
4. **颜色模式文案两套命名混用**：衣橱编辑现有 `单色/主色+点缀色/多色/拼色`，衣橱 intake 现有 `单主色/拼色/主辅色`。v2 阶段 1 决定用哪套，**本稿建议用 intake 命名**（更直接、与 P1/P2 一致），阶段 2 把编辑页命名同步过来。
5. **"需要留意"删除的回归风险**：v1.1.23 标签收紧计划已声明删除，但实际是否同步删除了 grep 7 个 `wishlist-intake-flow.tsx` 的测试？阶段 1 实施前先 `grep -rn "需要留意" src/` 确认所有引用都已清理。
6. **"评估信息"卡位置**：v2 明确归 P5 详情页独有。如果阶段 2 抽 `wishlist-extras.tsx` 误把评估信息塞进编辑页，需要在 PR review 中强制拒绝。
7. **横屏适配**：6 个页面都有横屏风险。`item/row.tsx` / `item/field.tsx` 必须支持 `sm:grid-cols-2`；chip 行在 `sm` 断点以上允许 4 列/行。
8. **版型/颜色模式中文 label 函数**：需要在 `src/lib/format.ts`（或新增 `src/lib/label.ts`）新建 `formatFitGender(value: GarmentFitGender | undefined): string` 和 `formatColorMode(value: ColorMode | undefined): string`，避免 6 个页面重复写 switch-case。
